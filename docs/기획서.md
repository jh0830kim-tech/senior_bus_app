**문서 버전:** v0.2 (자주 가는 장소, 탑승 안내 기능 추가)
**작성일:** 2026-07-20
**배포 형태:** 모바일 앱 (Android / iOS)
**버전 관리:** GitHub

## 1. 프로젝트 개요

### 1.1 배경 및 목적

고령층은 스마트폰 기반 대중교통 앱(네이버지도, 카카오맵 등)의 복잡한 화면 구성과 작은 글씨 때문에 버스 정보를 확인하는 데 큰 어려움을 겪는다. 본 프로젝트는 **어르신이 버튼 하나로 가장 가까운 정류장과 버스 노선을 확인**할 수 있는, 극도로 단순하고 직관적인 버스 정보 앱을 개발하는 것을 목적으로 한다.

### 1.2 핵심 가치

| 가치 | 설명 |
|---|---|
| **단순함** | 한 화면 = 한 가지 기능. 검색 버튼 하나로 시작 |
| **가독성** | 큰 글씨, 높은 명도 대비, 큰 터치 영역 |
| **안심** | 막힐 때 바로 사람과 연결되는 전화 서비스콜 버튼 |
| **확장성** | 모듈형 설계로 기능 추가(택시 호출, 지하철, 음성 안내 등)가 쉬운 구조 |

### 1.3 타겟 사용자

- 주 사용자: 만 65세 이상 고령층 (스마트폰 기본 조작 가능 수준)
- 보조 사용자: 어르신의 보호자(자녀), 지자체 복지 담당자

## 2. 핵심 기능 정의 (요구사항)

### F1. 내 주변 정류장 찾기
- 홈 화면의 **대형 검색 버튼 1개**를 누르면 현재 위치(GPS) 기준 **가장 가까운 정류장**을 표시
- 정류장까지의 거리(도보 시간)와 방향을 큰 글씨로 안내
- 해당 정류장을 지나는 **버스 노선 목록 + 실시간 도착 정보** 표시

### F2. 목적지 길찾기 (2가지 모드)
- 목적지 입력(음성 입력 지원 권장) 후 두 가지 결과를 **탭 2개로 단순 구분**하여 제공
  1. **빠른 길** — 환승 포함 최단시간 노선
  2. **편한 길** — 환승 없는 직행 버스 노선 (없으면 "환승 없는 버스가 없어요" 안내)
- 결과 화면: "① 몇 번 버스를 → ② 어디서 타고 → ③ 어디서 내리세요" 3단계 카드 형식

### F3. 시니어 친화 UI/UX
- 기본 글자 크기 20pt 이상, 시스템 글자 크기 설정 연동(최대 200% 확대)
- 버튼 최소 터치 영역 64dp 이상
- 색상 대비 WCAG AA 이상 (명도 대비 4.5:1 이상)
- 화면당 선택지 3개 이하, 뒤로가기 항상 동일 위치
- 전문 용어 배제 ("승차" → "타기", "하차" → "내리기")

### F4. 전화 서비스콜
- 모든 화면 하단에 **고정된 전화 버튼** 배치
- 누르면 콜센터(또는 지정 보호자 번호)로 즉시 전화 연결
- 향후 확장: 상담원이 어르신의 현재 위치를 조회할 수 있는 관리자 웹 연동

### F5. 자주 가는 장소
- 홈 화면에 **"자주 가는 곳" 대형 버튼** 배치
- 장소 등록: 이름(예: "딸네 집", "병원", "시장") + 위치를 저장 (최대 6곳 권장 — 화면당 큰 버튼으로 다 보이는 개수)
- 장소 삭제: 목록에서 길게 누르거나 "지우기" 버튼 → "정말 지울까요?" 큰 글씨 확인 창
- 등록된 장소를 누르면 **목적지 입력 없이 바로 길찾기(F2) 실행** → 어르신이 글자를 입력할 일 자체를 줄이는 것이 핵심
- 저장은 기기 내부(로컬)에 보관, 향후 보호자 앱에서 대신 등록해주는 기능으로 확장 가능

### F6. 탑승 안내 (노선 선택 후 실시간 안내)
길찾기 결과에서 어르신이 노선 하나를 선택하면, 타고 → 갈아타고 → 내릴 때까지 단계별로 안내한다.
1. **대기 안내** — "1234번 버스가 **5분 뒤** 도착해요" (실시간 도착 정보, 30초마다 갱신)
2. **탑승 확인** — 버스 도착 시각에 맞춰 "지금 오는 버스를 타세요" 표시
3. **환승 안내** — 환승이 있는 노선이면 "○○정류장에서 내려서 → △△번 버스로 갈아타세요" + 환승 정류장 접근 시(GPS 기준) 화면·진동으로 미리 알림
4. **하차 안내** — 내릴 정류장 1~2개 전부터 "다음다음 정류장에서 내리세요" → "**다음 정류장에서 내리세요**" 큰 글씨 + 진동 알림
- 안내는 한 화면에 **현재 해야 할 일 한 가지만** 표시 (지도보다 문장 중심)
- 향후 확장: 음성 안내(TTS)로 화면을 보지 않아도 되도록 발전

## 3. 화면 흐름 (초안)

```
[홈 화면]
 ├─ 🔍 대형 버튼: "가까운 정류장 찾기"  → [정류장 결과 화면]
 ├─ 🚌 대형 버튼: "버스로 길찾기"       → [목적지 입력 화면]
 ├─ ⭐ 대형 버튼: "자주 가는 곳"        → [자주 가는 곳 화면]
 └─ 📞 하단 고정: "전화로 물어보기"     → 전화 연결

[정류장 결과 화면]
 ├─ 정류장 이름 / 거리 / 방향 (큰 글씨)
 └─ 버스 노선 목록 + 도착까지 남은 시간

[자주 가는 곳 화면]
 ├─ 등록된 장소 대형 버튼 목록 (누르면 바로 길찾기 실행)
 ├─ ➕ "새 장소 넣기"  → 이름 + 위치 등록
 └─ ➖ "지우기" → 확인 창 후 삭제

[목적지 입력 화면] (음성/텍스트, 자주 가는 곳에서 오면 건너뜀)
 └→ [길찾기 결과 화면]
      ├─ 탭1: 빠른 길 (환승 포함 최단시간)
      └─ 탭2: 편한 길 (환승 없음)
           └─ 노선 선택 → [탑승 안내 화면]

[탑승 안내 화면]  ※ 한 번에 한 가지 안내만 표시
 ① 대기: "1234번, 5분 뒤 도착"
 ② 환승: "○○에서 내려 △△번으로 갈아타세요" (+진동)
 ③ 하차: "다음 정류장에서 내리세요" (+진동)
```

## 4. 시스템 아키텍처 (모듈형 설계)

### 4.1 설계 원칙

1. **레이어드 아키텍처 + 기능별 모듈 분리** — 화면(Presentation), 비즈니스 로직(Domain), 데이터(Data)를 분리하고, 기능(Feature) 단위로 패키지를 나눈다.
2. **인터페이스 기반 의존성** — 외부 API(버스 정보, 지도, 길찾기)는 반드시 추상 인터페이스(Repository)를 통해 접근한다. → API 공급자 교체·추가가 쉬움.
3. **Core 모듈 공유** — UI 컴포넌트(큰 버튼, 큰 글씨 텍스트), 상수, 유틸은 core 모듈에 두어 모든 기능 모듈이 재사용한다.
4. **신규 기능 = 신규 feature 모듈** — 향후 "택시 호출", "지하철", "음성 안내" 등은 features/ 아래 폴더 추가만으로 확장한다.

### 4.2 권장 기술 스택

| 구분 | 선택 | 이유 |
|---|---|---|
| 프레임워크 | **Flutter (Dart)** | 하나의 코드로 Android/iOS 동시 배포, 글자 크기·접근성 제어 용이 |
| 상태 관리 | Riverpod (또는 Bloc) | 모듈 간 결합도 낮은 상태 관리 |
| 지도 | 네이버/카카오 지도 SDK | 국내 정확도 |
| 버스 실시간 정보 | 공공데이터포털 TAGO API (국토교통부) 또는 각 지자체 버스정보시스템(BIS) API | 무료, 전국 정류장·도착 정보 |
| 길찾기(환승 포함) | ODsay 대중교통 API (또는 Tmap 대중교통 API) | 환승 포함 최단시간 경로 제공 |
| 로컬 저장 | shared_preferences / Hive | 자주 가는 곳, 글자 크기 설정 저장 |

> API는 모두 Repository 인터페이스 뒤에 두므로, 추후 공급자를 바꿔도 앱 코드는 수정이 최소화된다.

### 4.4 모듈 간 의존 방향

```
features  →  domain(usecases, 인터페이스)  ←구현─  data
    ↘            ↓
      core (누구나 사용 가능, 아무것도 의존하지 않음)
```

- **features는 data를 직접 알지 못한다.** 반드시 domain의 인터페이스를 통해서만 데이터를 요청 → API를 바꿔도 화면 코드는 그대로.
- 새 기능 추가 시: `features/새기능/` + 필요하면 `domain/usecases/` 추가. 기존 코드 수정 최소화.

## 5. GitHub 버전 관리 전략

### 5.1 브랜치 전략 (단순화한 Git Flow)

| 브랜치 | 용도 |
|---|---|
| `main` | 스토어 배포 버전만 존재. 태그(v1.0.0)로 릴리스 표시 |
| `develop` | 다음 버전 개발 통합 브랜치 |
| `feature/기능명` | 기능 개발 (예: `feature/nearby-station`) |
| `fix/이슈번호` | 버그 수정 |

**흐름:** `feature/*` → Pull Request → `develop` → 안정화 후 → `main` 병합 + 버전 태그

### 5.2 버전 규칙 (Semantic Versioning)

- **v메이저.마이너.패치** (예: v1.2.3)
  - 메이저: 대규모 개편
  - 마이너: 기능 추가 (예: 음성 안내 추가)
  - 패치: 버그 수정

### 5.3 협업 규칙

- 커밋 메시지 규칙: `feat:`, `fix:`, `docs:`, `refactor:` 접두어 사용
- 모든 병합은 Pull Request로 진행 (셀프 머지 지양)
- GitHub Issues로 작업 관리, GitHub Actions(ci.yml)로 자동 빌드·테스트

## 6. 개발 로드맵

| 단계 | 기간(안) | 내용 |
|---|---|---|
| **1단계 MVP** | 1~2개월 | F1(가까운 정류장) + F3(큰 글씨 UI) + F4(전화 버튼) + F5(자주 가는 곳), Android 우선 |
| **2단계** | +1~2개월 | F2(빠른 길/편한 길 길찾기) + F6(탑승 안내: 대기·환승·하차), 음성 입력, iOS 대응 |
| **3단계 배포** | +1개월 | 어르신 사용성 테스트(필수!) → 개선 → Google Play / App Store 출시 |
| **4단계 확장** | 이후 | 음성 안내(TTS), 보호자 연동, 택시 호출 등 feature 모듈 추가 |

## 7. 리스크 및 고려사항

1. **API 발급** — TAGO·ODsay API는 사전 키 발급 필요. 호출량 제한 확인(무료 티어 일 1,000건 수준) 후 필요 시 지자체 BIS 협의.
2. **위치 권한** — 어르신이 권한 팝업에서 막히지 않도록, 그림과 쉬운 말로 된 권한 안내 화면 필수.
3. **사용성 테스트** — 반드시 실제 어르신 5명 이상 대상 테스트를 배포 전 진행. "버튼을 찾는 데 3초 이상 걸리면 실패"를 기준으로 개선.
4. **오프라인 대응** — 통신 실패 시 "인터넷이 잘 안 돼요. 전화 버튼을 눌러주세요" 같은 쉬운 안내로 서비스콜 유도.
5. **iOS 전화 연결** — 전화 걸기 기능은 양 플랫폼 정책 확인 필요 (tel: 스킴 사용).
6. **탑승 안내 배터리/백그라운드** — F6 실시간 안내는 GPS 추적 + 주기적 API 호출이 필요. 백그라운드 위치 권한(안드로이드/ iOS 정책 상이)과 배터리 소모를 고려해, 탑승 안내 중에만 추적을 켜고 종료 시 즉시 끄는 설계 필수.
7. **GPS 오차 대응** — 지하·터널 구간에서 위치가 튈 수 있으므로, 하차 안내는 GPS 단독이 아니라 "정류장 통과 순서(도착정보 API)"와 교차 확인하는 방식 권장.

## 8. 부록 — 확장 시나리오 예시

**"음성 안내" 기능을 추가한다면?**
1. `features/voice_guide/` 폴더 생성
2. `domain/usecases/speak_route.dart` 추가
3. `core/ui`의 기존 카드 위젯에 스피커 버튼만 연결
→ 기존 코드는 거의 손대지 않고 기능 추가 완료. 이것이 본 기획서가 제안하는 모듈형 설계의 목적이다.
